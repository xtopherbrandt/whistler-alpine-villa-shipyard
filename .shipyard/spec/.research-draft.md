## Research Findings

### Auth library (HIGH confidence)
**Use Auth.js v5 (NextAuth)** with credentials provider.
- App Router native in v5 — `auth()` call works in server components and middleware
- Handles 30-day rolling sessions via httpOnly cookie out of the box
- Credentials provider for email+password fits the invitation model
- Password reset is NOT built-in for credentials provider — requires a custom token table in DB
- Install: `npm install next-auth@beta`

### Database (CRITICAL — not yet decided)
All three E005 features require persistent storage: users, roles, unit assignments, sessions, invite tokens, password reset tokens. The database choice must be made before these features can be built.
- **Recommendation: PostgreSQL via Prisma ORM** — relational model fits the cooperative data (users → roles → units → occupancy records). Prisma provides type-safe queries, migrations, and a schema file.
- Auth.js has a Prisma adapter that handles the session/account tables automatically.
- Supabase is an alternative (PostgreSQL + auth built-in) but introduces vendor lock-in.

### Email service (HIGH confidence)
**Use Resend** for invite emails and password reset emails.
- Modern Next.js-friendly API (Server Action compatible)
- Free tier: 3,000 emails/month (vastly more than needed for ~70 users)
- `npm install resend`
- Alternative: SendGrid (100/day free, more established but more complex)

### Invite link security (HIGH confidence)
- Token: `crypto.randomBytes(32).toString('hex')` — 256-bit cryptographic random
- Expiry: 72 hours (balances usability and security — 3 days for someone to check email)
- Single-use: token invalidated on first use (mark as `used: true` in DB)
- Store as bcrypt hash? No — these are short-lived tokens, store as-is with expiry

### Password handling (HIGH confidence)
- Hash with **bcryptjs** (pure JS, edge-compatible, constant-time comparison)
- Minimum length: 8 characters (NIST SP 800-63B guidance — length is the most effective requirement)
- No complexity requirements (uppercase, numbers, etc.) — NIST explicitly recommends against them
- bcrypt has a 72-byte input limit — trim or reject passwords > 72 chars (rare but needs handling)

### Rate limiting (HIGH confidence)
Login attempts must be rate-limited to prevent brute force.
- **5 failed attempts per 10-minute window = 15-minute lockout**
- Implement in the login Server Action using an in-memory store or DB counter
- For a small app: use Vercel KV or simple DB-backed attempt counter
- Return generic "Invalid email or password" regardless of which field is wrong (prevents email enumeration)

### Session security (HIGH confidence)
- httpOnly, Secure, SameSite=Lax cookies (Lax allows CSRF-safe GET-initiated sessions)
- 30-day rolling expiry — renewed on each successful request
- Auth.js handles this automatically

## Challenge Resolutions

- **Database:** PostgreSQL + Prisma ORM. Cross-epic concern — affects all features.
- **Auth library:** Auth.js v5 (NextAuth) with credentials provider.
- **Email service:** Resend (`npm install resend`).
- **Rate limiting:** 5 failed logins per 10-min window → 15-min lockout. Implemented in login Server Action.
- **Invite expiry:** 72 hours. Single-use token stored in DB. Admin can resend.
- **Password policy:** Min 8 chars, max 72 chars, length only (NIST SP 800-63B). bcryptjs for hashing.
- **Sessions:** 30-day rolling, httpOnly cookies via Auth.js.
